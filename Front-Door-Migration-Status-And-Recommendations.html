Front Door Classic-to-Standard Migration

Status report and recommended next steps · Prepared by Syed Rizvi · 2026-05-07
Summary: The maintenance window was used to migrate the five Front Door classic profiles to Standard tier. The migration was attempted via Azure REST API; Microsoft's migration service hard-blocks the Standard tier for these specific profiles due to managed WAF rule sets attached at the source. No production change occurred; classic profiles continue to serve traffic exactly as before. A decision is needed on which of two paths to take before re-running the migration.

2. What was attempted

A migration script was prepared and tested end-to-end against the documented Azure migration REST API (Microsoft.Cdn 2025-04-15):

  • Discovery of all five classic profiles via Azure REST
  • Identification of all Web Application Firewall (WAF) policies linked to each classic Front Door
  • Pre-creation of Standard-tier WAF policy copies (custom rules preserved, managed rules omitted — Standard tier does not support managed rule sets)
  • Submission of migration request including the migrationWebApplicationFirewallMappings body element to map each classic WAF to its new Standard-tier counterpart
  • Asynchronous polling of the azure-asyncoperation endpoint to capture the real backend operation status (not just the initial HTTP 202 acknowledgement)

4. Why Standard tier is blocked

Microsoft API response (verbatim, captured from the async operation result):
{"code":"BadRequest","message":"This frontdoor can only be migrated to premium sku"}

The Azure migration engine evaluates the source classic WAF policies, not the destination. Each of the four Front Door classic profiles has at least one WAF policy attached that contains managed rule sets (Default Rule Set, Bot Manager, etc.). When managed rule sets are present at the source, Microsoft's migration service forces the destination to Premium tier regardless of what mapping is supplied in the request body. Providing a Standard-tier WAF policy as the migration target does not override this check.

Reference (Microsoft documentation):

  • Front Door (classic) to Standard / Premium tier migration — explicitly states that profiles with managed WAF rules default to Premium during migration and cannot be downgraded to Standard later.

Important downgrade constraint

Premium-to-Standard downgrade is not supported. Microsoft's policy: a Standard profile can be upgraded to Premium after migration, but a Premium profile cannot be downgraded to Standard. The SKU choice made at migration time is permanent for that profile's lifetime.

4. The two viable paths forward

Option A — Migrate to Premium tier

What it means: Accept Microsoft's mandate. Migrate the four Front Door classics to Front Door Premium. The classic WAF policies (with their managed rule sets) carry over fully. CDN profile standard still goes to Standard tier (no managed WAF on CDN side).

Pros:

  • Works in a single maintenance window (~30 minutes total)
  • Managed WAF rule protections are preserved exactly as today
  • Zero security trade-off
  • Premium feature set unlocked (Private Link, advanced rules engine, full diagnostics)

Cons:

  • Approximate cost increase: ~$1,200–$1,400 per month total across the four profiles (Standard ~$35/mo each vs Premium ~$330/mo each, base only; bandwidth and request charges similar between tiers)
  • Permanent — cannot be downgraded later
Verdict: Lowest-risk, fastest path. Recommended if budget for ~$1,300/month uplift is acceptable.

Option B — Strip managed rules, then migrate to Standard

What it means: Edit each classic WAF policy to remove its managed rule sets before running the migration. Once classic WAFs contain only custom rules, Microsoft's migration engine permits Standard tier. Migrate, then optionally re-implement equivalents of the managed rules as custom rules on the Standard-tier WAF policies.

Pros:

  • Significant ongoing cost saving (~$1,300/month)
  • Matches existing scale — Standard tier is sufficient for the current configuration

Cons:

  • Requires a controlled change to the production WAF policies (managed rule removal). During the window between rule removal and full migration, those managed protections (DRS, Bot Manager) are not active.
  • Needs a separate change window plus security sign-off
  • Custom-rule equivalents for previously-managed protections may require additional engineering and ongoing maintenance
  • Slightly higher rollback complexity if migration fails after rules are stripped
Verdict: Best long-term cost outcome but requires explicit security sign-off on temporarily reduced protection during the migration window.

7. Detailed implementation plan

Option A — Implementation steps (Premium)

  1. Pre-flight (15 min, business hours OK):
    • Confirm budget approval for the SKU uplift
    • Snapshot the existing AFD classic configurations (already captured)
    • Confirm DNS owner is on standby for any post-migration CNAME updates if custom domains are present
  2. Maintenance window (30–45 min):
    • Run the prepared migration script with the Premium SKU and the WAF mappings already pre-staged
    • Per profile: POST /providers/Microsoft.Cdn/migrate with {sku: Premium_AzureFrontDoor, classicResourceReference, profileName, migrationWebApplicationFirewallMappings}
    • Poll the azure-asyncoperation URL until status=Succeeded per profile (typically 5–10 minutes per profile)
    • Then per profile: POST /providers/Microsoft.Cdn/profiles/{newName}/migrationCommit
    • Verify each new profile via GET: SKU is Premium, provisioningState=Succeeded
    • CDN classic standard profile uses the parallel CDN path (cdnMigrateToAfd) and goes to Standard tier (no managed WAF on CDN side)
  3. Post-migration (within 24h):
    • Optionally update DNS records to point custom domains at the new *.azurefd.net endpoints
    • Classic profiles auto-cleanup after approximately 15 days
    • HTML report delivered with verified state per profile

Option B — Implementation steps (Standard, requires WAF rule strip first)

  1. Pre-flight (separate session, security sign-off required):
    • Inventory managed rule sets currently active on each classic WAF (Default Rule Set version, Bot Manager, etc.)
    • Decide for each managed rule: keep as custom-rule equivalent on the new Standard WAF, drop entirely, or accept temporary loss
    • Obtain security sign-off for the protection delta during the migration window
    • Back up each classic WAF policy JSON before modification
  2. Step 1 of the maintenance window: strip managed rules (~10 min):
    • For each of the four classic WAF policies, PATCH the policy to remove the managedRules block (or set managedRules.managedRuleSets to empty)
    • Wait ~2 minutes for the WAF change to propagate edge-side
    • During this window, only custom rules are protecting traffic on those four classic AFDs
  3. Step 2 of the window: migrate (~30–45 min):
    • Run the prepared migration script with Standard SKU
    • The pre-staged Standard-tier WAF policies (PyxIQPolicyStd, wafPyxPwaProdStd, PyxIQPolicyStageStd, wafPyxPwaStageStd) are mapped in the migrate request body
    • Same async polling and commit flow as Option A
  4. Post-migration (within 24h):
    • Optionally re-implement equivalents of the previously-managed rules as custom rules on the new Standard WAF policies
    • HTML report with verified state per profile

9. Recommendation

Decision needed on Option A vs Option B before the next maintenance window.

  1. If the ~$14K/year uplift is acceptable, go with Option A (Premium) — one maintenance window of ~30 minutes, no security trade-off, managed WAF protections are preserved.
  2. If the cost is not acceptable, go with Option B (Standard after managed-rule removal) — saves ~$14K/year but requires explicit security sign-off, separate planning to inventory managed rules in use today, and a longer maintenance window with a brief protection-reduced phase before migration starts.

Either path can be executed against the prepared script and the same maintenance window mechanics. The only delta between the two is whether managed rules are stripped from the source classic WAFs first.

10. Risk and rollback posture (today)

  • All five classic profiles are fully operational. Migration attempts failed at the Azure control-plane validation step before any change was made to a profile.
  • No DNS records were modified.
  • No certificates were re-issued or changed.
  • No traffic-affecting changes occurred during the maintenance window.
  • Standard-tier WAF policies were pre-staged in the same resource groups as the originals; they are inert until referenced by a successful migration. They can be safely deleted or left in place pending a decision.
