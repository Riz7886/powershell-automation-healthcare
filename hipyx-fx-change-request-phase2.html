1. Background and prior approval

The hipyx Front Door profile in the production resource group is on the Azure Front Door Classic SKU. Microsoft is retiring AFD Classic on March 31, 2027. Managed certificates on Classic profiles begin auto-migrating starting April 10, 2026 (best effort) and Classic-managed certs expire April 14, 2026. To keep ownership of the migration on our schedule rather than Microsoft's, the migration was approved as a phased move per domain on the hipyx Classic profile to a new Standard profile (hipyx-std).

Phase 1 was approved and executed on 2026-04-24: survey.farmboxrx.com moved cleanly from Classic hipyx to Standard hipyx-std, with WAF policy hipyxWafPolicy, managed cert in Approved state, and DNS handoff completed. Phase 2 (www.hipyx.com) was deferred from that window for risk reasons described below.

2. Why phased - and why www was not in Phase 1

Running both domains in a single change window would have meant any failure on the second domain would extend the outage window for the entire profile. Phase 1 took the lower-traffic non-customer-facing domain first specifically so:

  • The migration pattern (Classic detach -> Standard create -> WAF bind -> DNS handoff) could be validated end-to-end on a lower-impact domain.
  • The Standard SKU's WAF behavior (hipyxWafPolicy, Detection mode, custom rules - Standard SKU does not support managed rules) was confirmed in production.
  • The DNS handoff workflow with the DNS owner was rehearsed.
  • If anything went wrong during Phase 1, www.hipyx.com was guaranteed unaffected.
  • The runbook (section 6) and rollback procedure (section 9) were finalized using Phase 1 as the template.

Phase 1 succeeded. The pattern is validated. Phase 2 now uses that validated pattern, with the additional safeguards listed in this CR.

6. Cutover runbook (T-0)

The cutover is automated end-to-end by hipyx-cutover-tonight.ps1. The script handles: detach from Classic routing rules, delete Classic frontend-endpoint, wait for Azure hostname release (10 min), retry create on Standard up to 8 times with 5-min waits (40-min budget), build Standard endpoint + custom-domain + origin group + origin + route + WAF binding, output the DNS handoff package as both text and HTML.

If all 8 attempts fail (Azure-side issue beyond the 40-min budget), the script auto-rolls back to Classic and exits cleanly. No partial state.

One command to start the cutover:

cd C:\path\to\PYX-scripts
.\hipyx-cutover-tonight.ps1

Defaults baked in (no parameters required for the standard cutover):

Subscriptione42e94b5-c6f8-4af0-a41b-16fda520de6e Resource groupproduction Classic profilehipyx Standard profilehipyx-std Hostnamewww.hipyx.com Origin hostappsvc-pwa-prod.azurewebsites.net WAF policyhipyxWafPolicy Hostname release wait600 seconds (10 min) Retry wait between attempts300 seconds (5 min) Max attempts8

The script prompts for explicit YES confirmation before doing anything destructive. Pass -NoConfirm to skip the prompt during the maintenance window if desired.

What the script does, step by step:

  1. Pre-flight - confirms az CLI signed in, sets subscription, ensures front-door extension is current.
  2. Confirmation gate - prompts YES to proceed (skippable with -NoConfirm).
  3. Phase A.1 - Detach from Classic - removes www-hipyx-com reference from httpToHttpsRedirect and defaultForwardingRule Classic routing rules.
  4. Phase A.2 - Release the hostname - deletes the Classic frontend-endpoint www-hipyx-com; waits 10 minutes for Azure to release the global hostname binding.
  5. Phase A.3 - Create on Standard - up to 8 attempts spaced 5 minutes apart to create the custom-domain on hipyx-std with managed-cert and TLS 1.2 minimum. Total budget: 40 minutes.
  6. Phase A.4 - Build the rest - Standard endpoint pyx-fx-www-hipyx-com-ep, origin group og-www-hipyx-com, origin pointing at appsvc-pwa-prod.azurewebsites.net (Host header preserved), route binding the custom-domain, security policy binding hipyxWafPolicy.
  7. Phase A.5 - DNS handoff - script writes dns-handoff-<ts>.html and .txt to the Desktop report folder. The HTML is the file emailed to the DNS owner.
  8. Phase B (only if Phase A fails) - automatic rollback. Recreates Classic frontend-endpoint, re-attaches to both routing rules, re-enables HTTPS managed cert. www.hipyx.com resumes on Classic AFD within 1-2 min.
Outage window: The user-visible outage is from Phase A.2 (Classic frontend-endpoint deletion) until either (a) the DNS owner publishes the new CNAME after cert is Approved, or (b) the rollback completes and Classic resumes serving. Expected 5-15 min real-time.

7. DNS handoff procedure

When the script finishes successfully, two DNS records are produced. The DNS owner publishes them in this order:

StepTypeHostValueTTLNotes
1TXT_dnsauth.www(provided by script - validation token)300Publish first. Validates the AFD Standard managed cert.
2WaitWait until cert state shows Approved (5-30 min)-
3CNAMEwww(provided by script - new endpoint hostname, e.g. pyx-fx-www-hipyx-com-ep-XXXX.z02.azurefd.net)300Publish AFTER cert state is Approved. This is the cutover.

The DNS owner watches the cert state with this command (or the executor does and signals to the DNS owner):

az afd custom-domain show -g production --profile-name hipyx-std --custom-domain-name www-hipyx-com --query domainValidationState -o tsv

States cycle: Pending -> Submitting -> Approved.

8. Verification commands

After the CNAME is published and propagated (1-2 min on TTL=300):

nslookup www.hipyx.com
curl.exe -I https://www.hipyx.com/

Expected:

  • nslookup resolves www.hipyx.com through *.z02.azurefd.net (Standard) - no longer through hipyx.azurefd.net (Classic).
  • curl -I returns HTTP 200 (or appropriate app response) with the response header x-azure-ref.
  • Application smoke test on www.hipyx.com matches pre-cutover behavior (login, key user flows).

Continuous polling (auto-detects when stable):

.\verify-www-hipyx.ps1

Polls HTTP every 20 sec, declares green after 3 consecutive 2xx/3xx responses. Times out at 30 min and recommends Microsoft support escalation if still red.

9. Rollback plan and commands

Rollback is DNS-level, validated during Phase 1 dry-run, and takes under 5 minutes including DNS propagation. Two scenarios:

9.1 Rollback during the cutover (script auto-rollback)

If hipyx-cutover-tonight.ps1 exhausts all 8 Standard create attempts, it automatically falls into Phase B and:

  1. Recreates the Classic frontend-endpoint www-hipyx-com.
  2. Re-attaches it to both Classic routing rules.
  3. Re-enables HTTPS managed cert on Classic.
  4. www.hipyx.com resumes on Classic AFD within 1-2 min. DNS does not change in this scenario.

9.2 Manual rollback after the CNAME flip

If verification fails after the DNS owner has already flipped the CNAME, the DNS owner repoints the CNAME back:

www  CNAME  hipyx.azurefd.net  TTL 300

Within the 300-second TTL, traffic returns to Classic AFD. The Standard configuration is left in place for the retry; only DNS has changed.

9.3 Standalone rollback script

For manual invocation outside the cutover script:

.\hipyx-cutover-tonight.ps1 -ForceRollback -NoConfirm

or the dedicated script:

.\rollback-www-hipyx.ps1

Either restores the Classic frontend-endpoint and routing rule references. Idempotent - safe to re-run.
