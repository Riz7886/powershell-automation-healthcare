PYX Front Door Migration - Status and Recommendation

Production cutover window | April 29-30, 2026

Summary: The migration window opened the evening of April 29 with a clear plan to move the five remaining PYX Front Door Classic profiles to the new Azure Front Door Standard SKU. After working through the migration tooling for several hours, the Microsoft Azure portal itself surfaced a platform constraint that forces the production profiles onto Premium tier rather than Standard. Rather than make a unilateral cost or security call on production traffic, I have paused at the decision point and brought this back for guidance.

1. What I did tonight

The plan was to migrate hipyx, pyxiq, pyxiq-stage, pyxpwa-stage, and the standard CDN profile from Classic to the new SKU before the Microsoft retirement deadline of March 31, 2027.

I built a PowerShell migration script earlier in the day and worked through it across the evening, refining the WAF mapping logic and the cmdlet parameter format as each test surfaced a new aspect of the underlying constraint. Approaches exercised against the live environment:

  • Azure CLI (az afd profile migrate) with the documented WAF mapping arguments
  • Direct Azure REST API calls against the migration endpoint
  • Native Azure PowerShell (Az.Cdn) module cmdlets - the Microsoft-recommended path
  • Pre-creating new Standard-tier WAF policies and explicitly mapping them during migration
  • Azure portal Migration wizard - the interactive Microsoft-supplied tool

Every approach surfaced the same root cause at the WAF mapping step. The constraint is enforced by Microsoft at the Azure control-plane layer, not at the tooling layer. Once the Azure portal Migration wizard surfaced the same error visibly and explicitly, the picture was complete - this is a Microsoft platform constraint, and it requires a tier decision rather than more tooling work.

2. Why it did not work - the Microsoft constraint

The Microsoft Azure portal Migration wizard greys out the Standard tier option for the hipyx profile and displays the following message at the Prepare step:

The AFD (classic) instance has managed WAF rules associated, and will be migrated to Premium tier. If you want to migrate to Standard tier, please remove the association on AFD (classic) and migrate afterwards.

Translation - the Classic Web Application Firewall policy on www.hipyx.com uses Microsoft-managed rule sets (OWASP protection, Bot Manager, Microsoft threat intelligence). The new Standard SKU does not support managed rule sets at all - only custom rules. The portal will not allow a downgrade to Standard while managed WAF rules are attached, because the Standard SKU physically cannot host them.

The only way to force Standard tier would be to first detach the managed WAF policy from the Classic profile, migrate, then author and attach a new Standard custom-rules-only WAF policy. That sequence leaves www.hipyx.com with no WAF protection at all for 20 to 45 minutes per profile. For a production login domain, that exposure window is not something I am willing to accept on my own without explicit sign-off.

3. What I recommend we do

Two Premium, three Standard.
  • Premium tier ($330 per month each) for hipyx and pyxiq - production login traffic, protected by managed WAF rules today. Downgrading these is a real and permanent security regression.
  • Standard tier ($35 per month each) for pyxiq-stage, pyxpwa-stage, and the standard CDN profile - lower-risk surfaces, internal or staging use, custom rules sufficient.
This split preserves managed WAF coverage on the only two domains exposed to live customer login traffic, while taking the cost win on three lower-risk profiles.

4. Decision needed to proceed

One of the following before further work proceeds:
  1. Approve the recommended split - 2 Premium plus 3 Standard, $765 per month total. My recommendation.
  2. Approve all Premium - $1,650 per month. Simplest, full WAF coverage everywhere.
  3. Approve all Standard with documented WAF gap - $175 per month. Detach managed WAF on production profiles, migrate, reattach Standard custom-rules-only WAF afterwards. 20 to 45 minute WAF coverage gap per production profile is documented and accepted.
  4. Postpone - reschedule to a different window. Microsoft retirement deadline of March 31, 2027 still applies.

Once a tier is approved, I can execute the migration through the Azure portal Migration wizard - approximately 10 to 15 minutes per profile, plus 5 to 30 minutes per profile for DNS propagation. The DNS owner is already on standby. The wizard for hipyx is currently open at the Prepare step with the configuration ready to go.
