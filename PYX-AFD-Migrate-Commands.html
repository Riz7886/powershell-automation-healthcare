Step 4 - REAL migration with WAF detach + per-profile rollback gate (10 PM window)

What will happen per AFD profile (4 of them):
  1. Backup Classic WAF policy JSON to ~\Desktop\pyx-frontdoor-migration\snapshots-*\waf-pre-unlink-*.json
  2. Detach WAF from each FrontendEndpoint of the Classic profile (Set-AzFrontDoor with cleared link)
  3. Re-call Test-AzFrontDoorCdnProfileMigration - now returns DefaultSku=Standard_AzureFrontDoor
  4. Pre-create new Standard-tier WAF policies (no managed rules)
  5. Run Start-AzFrontDoorCdnProfilePrepareMigration with SkuName=Standard
  6. Pause for operator decision: COMMIT / ROLLBACK / SKIP
  7. If COMMIT: Enable-AzFrontDoorCdnProfileMigration flips traffic to Standard, retires Classic. Watchdog runs 5 min HEAD probes.
  8. If ROLLBACK: Stop-AzFrontDoorCdnProfileMigration deletes the new Standard profile. Classic remains active and unchanged.
For the CDN standard profile (5th): no WAF involved, migrates straight to Standard via Move-AzCdnProfileToAFD, then same COMMIT/ROLLBACK gate.

Interactive (recommended for first run - per-profile gate, you control each one)

.\PYX-AFD-Migrate-v2.ps1 -StripManagedRulesToForceStandard

You'll get prompted to type YES first, then per profile prompted to type COMMIT, ROLLBACK, or SKIP.

Unattended (only if you want all 5 to commit in sequence with no prompts)

.\PYX-AFD-Migrate-v2.ps1 -StripManagedRulesToForceStandard -AutoCommit -NoConfirm

Manual rollback (worst case - if Commit succeeded but you need to revert)

Once Commit fires, Classic is gone. Recovery requires re-creating a Standard profile from the Classic snapshot JSON in snapshots-*\. Do NOT commit if uncertain.

WAF re-attach post-migration

After successful Standard migration, attach a fresh Standard-tier WAF via Azure Portal:

  1. Open the new Standard profile (e.g. hipyx-std-v2) in Azure Portal
  2. Settings -> Security policies -> Add
  3. Create new WAF policy (Standard SKU) with the custom rules copied from the backup JSON
  4. Associate with the route
