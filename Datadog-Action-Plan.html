What still needs fixing

1Azure cloud integration broken (16 monitors)

All "auto-created" monitors using azure.* metrics. Last modified 2025-12-10 00:15 (single batch timestamp = single integration cutover that day). Includes: SQL Database CPU/Storage/DTU/Deadlocks, App Service CPU/Memory/Response Time/HTTP 5xx, NSG/RBAC/Privileged Role/Key Vault/Failed Auth audit monitors, PROD Network Packet Loss.

Root cause

Datadog's connection to Azure stopped working on 2025-12-10. Most likely the Service Principal client secret expired, or the SP lost RBAC permissions on the affected subscription, or the subscription mapping changed.

Steps to fix

  1. Open Datadog -> Integrations -> Azure
  2. For each configured tenant, expand its row and check the "Errors" indicator - any tenant showing red errors is the broken one
  3. For the broken tenant: confirm the Service Principal (client_id shown in Datadog) still exists in Azure Portal -> Microsoft Entra ID -> App Registrations
  4. If the SP exists but the client secret expired, generate a new client secret on the App Registration (Certificates & secrets -> New client secret -> 2 year expiry)
  5. In Datadog, click "Update Application" on the failing tenant and paste the new client secret
  6. Verify the SP still has "Monitoring Reader" role on every subscription you want Datadog to monitor: Azure Portal -> Subscription -> Access Control (IAM) -> Role assignments -> check the SP is listed with "Monitoring Reader"
  7. Wait 5-10 minutes - Datadog should start receiving azure.* metrics again
  8. Validate by querying any azure.* metric in Datadog Metrics Explorer and confirming data points are arriving
Owner: Whoever administers the Azure App Registrations + has Azure RBAC owner/admin on the affected subscriptions.

2Datadog Databricks integration broken (8 monitors)

All using databricks.* (official integration) metrics with tag hybrid:true env:prod service:databricks. Last modified 2025-12-10 11:04 (separate batch from cluster 1). Includes: DBX Cluster CPU/Memory/DBU Spike, DBX SQL Warehouse Latency High / Queries Failing, DBX Jobs Failing/Queued, DBX API 5xx Errors, DBX API Latency High.

Root cause

The official Datadog Databricks integration stopped publishing on 2025-12-10. Most likely the Databricks workspace token (PAT) expired or was revoked, or the Databricks system tables required by the integration were disabled in the workspace, or the integration was mistakenly removed from a workspace.

Steps to fix

  1. Open Datadog -> Integrations -> Databricks
  2. Verify each configured workspace shows green "Last seen" indicator within the last 5 minutes - any workspace showing older than that is broken
  3. For the broken workspace: in Databricks UI, generate a new Personal Access Token (User Settings -> Developer -> Access tokens -> Generate new token, lifetime 90 days or longer)
  4. Back in Datadog, click the workspace -> "Update workspace token" -> paste the new PAT
  5. In Databricks workspace UI, verify system tables are enabled: Settings -> Catalog -> system catalog -> ensure "compute", "billing", "access" schemas are enabled
  6. Wait 10-15 minutes for the next data fetch cycle
  7. Validate by querying databricks.cluster.cpu_percent in Datadog Metrics Explorer
Owner: Whoever has Databricks workspace admin rights + can generate Personal Access Tokens.

3Custom Databricks collector partially broken (5 monitors)

Monitors using custom.databricks.* namespace (not the official integration). Last modified 2025-12-11 14:17. Specific silent metrics: custom.databricks.api.5xx, custom.databricks.api.latency, custom.databricks.cluster.dbu, custom.databricks.jobs.failed, custom.databricks.sqlwarehouse.running.
Note: custom.databricks.cluster.cpu and custom.databricks.cluster.memory ARE still publishing - so the collector is partially running, just dropped some metric writes.

Root cause

Whoever owns the custom collector pushed a code change that removed or broke 5 specific metric write calls while leaving the others working. Common cause: refactoring metric names, exception swallowing in metric writes, or feature-flagging some metrics off.

Steps to fix

  1. Identify the custom collector. It is one of: a Databricks notebook on a scheduled job, an Airflow DAG that pushes via dogstatsd or the Datadog API, or a Python/script-based agent on a server. Search the codebase / Databricks workspace / Airflow for the string custom.databricks - that finds the metric writer.
  2. Once located, look at the recent code changes (git history or notebook revision history) for changes around 2025-12-11.
  3. Compare the current code against a working version - look for these 5 metric names and confirm they are still being written:
    • custom.databricks.api.5xx
    • custom.databricks.api.latency
    • custom.databricks.cluster.dbu
    • custom.databricks.jobs.failed
    • custom.databricks.sqlwarehouse.running
  4. Restore the missing metric writes. Verify the collector is also catching exceptions - silent failures here are how this happened.
  5. Re-run the collector once
  6. Validate in Datadog Metrics Explorer - all 5 metrics should be arriving within 15 minutes
Owner: Whoever maintains the custom Databricks metrics collector code.

Permanent prevention

  1. Long-life Service Principal secrets: use a dedicated Datadog Service Principal with a 2-year client secret, calendar-tracked for renewal 60 days before expiry. Avoid 90-day rotating secrets that lapse silently.
  2. Auto-monitor on the integrations themselves: add a Datadog "Cloud Integration Status" monitor or "Integration Heartbeat" check that alerts when an integration's last_seen exceeds 30 minutes. That way the next time this happens, Datadog tells us before the dashboard goes blank.
  3. Custom collector defensive coding: the silent metric writes in cluster 3 happened because exceptions were swallowed. Add an exception logger and a "metrics_emitted_count" counter so a missing metric write surfaces immediately.
