Section 1

What we are building, in one paragraph.

We are taking PYX Health's data, which today lives scattered across SQL Server, Salesforce, and FiveNine, and pulling it into one secure, governed home on Databricks running in Microsoft Azure. From that one home, the right people see the right data with the right protections. Tableau gets analytics-ready data. V4C gets the data-quality view they need. Engineers get raw data to work with. Patients' protected health information stays protected throughout. Auditors can see who looked at what, when.

Think of it this way

Today PYX's data is like having patient files in three different filing cabinets in three different offices. Some staff have keys to one cabinet, some to another, no one has a clean view across all three, and there's no central log of who opened which drawer. We are replacing that with one secure central building with a front desk, a guard at every floor, and a camera on every door.

Section 2

Why a "lakehouse" matters for PYX.

A "lakehouse" combines two older ideas. A "data warehouse" is a tidy library where everything is filed and searchable but expensive and slow to update. A "data lake" is a big warehouse floor where everything dumps unsorted but cheap and fast to add to. The lakehouse keeps the cheap, fast ingestion of a lake, with the tidy, searchable, secure access of a warehouse. Databricks is the leading lakehouse platform.

Why this matters specifically for PYX Health

  • HIPAA compliance audits get cleaner. Every query against PHI is logged with the user's identity. The audit log is built-in, not bolted on.
  • One source of truth for analytics. Today, Tony's reports might pull different numbers depending on which cabinet they sourced from. The lakehouse gives one canonical "Gold" copy that everyone agrees on.
  • Vendors plug in cleanly. V4C, Tableau, and any future analytics vendor get a scoped, audited connection. No more giving someone a database password and praying.
  • Cost is predictable. Budgets and alerts on every cost center. If somebody runs a query that would cost $5,000, the system catches it before it runs.
  • Development is safer. Engineers work against synthetic-data copies in DEV. No engineer ever touches real patient data unless they explicitly need to and have the right group membership.

Bottom line. The lakehouse moves PYX from "we have data in three places, hope nobody messes up" to "all our data is in one place, with audit-grade controls, ready to be analyzed without putting a single patient at risk."

Section 4

The three layers: Bronze, Silver, Gold.

The medallion model is industry standard for lakehouses. Data flows through three quality stages. Each stage adds polish and protection. Each stage has different access rules.

Kitchen analogy

Think of a restaurant. Bronze is the back loading dock where deliveries arrive in crates from suppliers. Raw, unlabeled, sometimes messy. Silver is the prep kitchen where ingredients are washed, chopped, and labeled. Some allergens are flagged and locked away. Gold is the plated dish that goes to the customer's table. Beautiful, ready to consume, no raw chicken in sight. The customer (Tableau, V4C) only ever sees the plate. Engineers work on the loading dock and prep kitchen.

Bronze layer

What it is: Raw data, exactly as it arrived from SQL Server, Salesforce, and FiveNine. Untouched, including all PHI.

Who can see it: Only the PYX data engineering team. No analyst, no scientist, no vendor, ever.

Why it exists: If a transformation later turns out to be wrong, we can go back to the original data and re-do it.

Silver layer

What it is: Cleaned, joined, conformed data. PHI is masked at this layer; row-level security limits who sees which organization's rows.

Who can see it: Engineers, data scientists, V4C (with masking), and admins.

Why it exists: The "single source of truth" that all reports build from.

Gold layer

What it is: Aggregated, business-ready data. No PHI by construction. Things like "patient counts by region by month."

Who can see it: Tableau, V4C, business analysts, plus all the upstream roles.

Why it exists: The only layer most consumers ever need to touch.

What that means in practice

If a Tableau dashboard breaks because someone deleted a Gold table by accident, no PHI was exposed because Tableau cannot see anything except Gold. If V4C's contract ends tomorrow, we revoke their service principal and they immediately lose all access; our internal teams keep working uninterrupted. If a regulator asks "show me everyone who queried the SSN column in the last 90 days," we run one query against the Unity Catalog audit log and produce a list.

Section 6

How PHI stays protected.

Two layers of protection apply automatically the moment anyone except an engineer queries patient data. The first hides specific fields. The second hides specific rows. Both are enforced by the platform itself; an engineer cannot accidentally bypass them in a report.

Layer 1: Column masking (hiding fields)

Five fields are automatically masked for everyone except engineers. The system runs the masking every time someone reads the data; the original values are still stored encrypted, but the reader only sees the masked version.

Censored letter analogy

Imagine a wartime letter. The recipient sees: "Dear ______, the operation is set for ______ at ______. Yours, ______." Same letter, same paper, but the sensitive parts are blacked out before the reader sees it. Our masking does the same thing in real time, every query, automatically.

Layer 2: Row-level security (hiding rows)

If V4C is contracted to monitor the data quality for one PYX customer organization, V4C should only see that customer's rows even if they query an entire shared table. The platform applies a filter automatically: "show me only rows where the organization-id matches the caller's assignment." A Tableau workbook for Customer A cannot accidentally surface Customer B's data; the filter blocks it.

Department-store analogy

Think of a department store's inventory system. The Macy's manager logs in and sees only Macy's inventory, even though the database holds inventory for Macy's, Bloomingdale's, and Nordstrom. The system filters by the manager's store-id automatically. Same database, isolated views.

What this means for HIPAA audits

An OCR auditor asking "prove a Tableau user couldn't see SSNs" gets a one-line answer: Tableau is a service principal with read access to Gold only; Gold contains no PHI by construction; even if it did, the masking function intercepts the query and returns the hashed value. That's three independent controls stacked. We can also produce, on demand, the full audit log of every query that touched the silver schema in the last 365 days, by user identity, with the exact statement.

Section 7

V4C and Tableau: the outside vendors.

Two external systems plug into the lakehouse. Each has its own scoped credential and its own scoped read access. Neither can see Bronze. Neither can write anything. Both leave an audit trail on every query.

V4C: data quality monitoring

What V4C does for PYX: Continuously checks the Silver layer for quality issues. Flags duplicate rows, missing values, schema drift, and invariants that should always hold (e.g., "patient age should be 0-120").

What V4C can read: Silver layer (with PHI masked and rows filtered to V4C's scope) and Gold layer (which has no PHI).

What V4C cannot read: Bronze in any form. Original PHI under any circumstance. Other organizations' rows.

How they connect: Through an Azure AD service principal we created for them, with a credential they hold. We can revoke it in seconds if the contract ends.

Tableau: leadership reporting

What Tableau does for PYX: Powers the dashboards Tony, Jack, and the leadership team look at. Counts, trends, KPIs.

What Tableau can read: Gold layer only. Aggregates, no PHI by design.

What Tableau cannot read: Silver, Bronze, or any source-specific schema. The system rejects the query before it runs.

How they connect: Through a separate Azure AD service principal scoped to Gold only. The Tableau infrastructure team never gets a database password.

Adding a future vendor

The pattern is the same for any future vendor (clinical analytics partner, billing audit firm, regulatory reporter): create a service principal, scope it to the exact schemas they need, and they can connect. Removing a vendor is one line: revoke the principal, the access is gone.

Section 8

The cost decision: Budget vs Enterprise.

We built two deployment profiles. Both contain the exact same governance: same access rules, same PHI masking, same vendor service principals, same audit trail. The difference is how much hardening we put on the underlying Azure infrastructure. The tradeoff is roughly $5,500 per month for additional defense in depth.

Budget profile

~$2,500

approximate monthly run rate, both DEV and PROD combined

What you get: Full Unity Catalog governance, all five PHI masks, row-level security, V4C and Tableau service principals, separate DEV and PROD environments, audit logs.

What you don't get: Customer-managed encryption key, geo-redundant storage, a dedicated firewall in front of Databricks, Microsoft Sentinel SIEM, 365-day log retention.

Enterprise profile

~$8,000

approximate monthly run rate, both DEV and PROD combined

What you get: Everything in Budget, plus customer-managed encryption key (PYX owns the key, rotates it every 90 days), geo-redundant storage, hub-spoke firewall, Microsoft Sentinel SIEM with HIPAA detection rules, 365-day audit retention, dedicated PHI pipeline scaffolding.

Why you'd want this: If we ever face a HIPAA audit, BAA negotiation, or a customer who wants to inspect our security posture, this is the answer that goes in the questionnaire and signs the deal.

Recommended hybrid (what we suggest). Run Budget for DEV (no real patient data lives there, so the heavy controls are wasted spend) and Enterprise for PROD (where real PHI lives and where audits happen). Combined monthly: roughly $4,500 to $5,500. That saves over $3,000 a month versus running Enterprise everywhere, while keeping production fully audit-ready.

Section 10

Frequently-asked questions.

Q. Why Databricks and not Snowflake or just SQL Server?

Snowflake and SQL Server are both fine for analytics, but neither was built for the lakehouse pattern of streaming data ingest plus structured warehouse on the same engine. Databricks runs both. It also has the strongest unity-catalog governance available, with PHI masking and row-level security built in rather than bolted on. For a HIPAA shop, that lowers audit risk meaningfully.

Q. What if Databricks goes down?

Source systems are unaffected; SQL Server, Salesforce, and FiveNine continue operating. Tableau dashboards built on Gold pause until the lakehouse is back. Critical operational reporting (the kind that runs the business) should not depend on the lakehouse alone; the lakehouse is for analytics, not transactional operations.

Q. How long does setup take?

Terraform apply takes about 35 minutes for a clean deploy. Most of that is waiting for Azure to provision the firewall (Enterprise profile) or the storage account replication. Re-applies after that are 2-3 minutes for governance changes.

Q. What if a vendor's contract ends?

One Terraform change removes their service principal. They lose access in seconds, the access keys still in their possession become inert. We can also rotate keys on a schedule if the contract requires it.

Q. What if we need to add a new data source?

For a new source like a clinical-trials database: add an Airflow DAG (or use Lakeflow Connect if it's a real-time source), add a new bronze schema (one Terraform line), and the existing access rules apply automatically. Engineers can write to it; nobody else can.

Q. Who has the customer-managed key?

In the Enterprise profile, Azure Key Vault holds the key. The key vault itself is access-controlled to a small Azure AD admin group (Tony, Syed, designated platform owners). Microsoft technicians, including Microsoft Support, do not have access to the key. The key auto-rotates every 90 days. If we ever needed to revoke all access at once, deleting the key effectively shreds the encrypted data.

Q. Can a single bad query take down the system?

No. Databricks SQL Warehouses have query timeouts and resource limits per workspace. A runaway query is killed automatically. Cost-alert rules trigger if daily spend crosses thresholds we set.

Q. What about backups?

Two layers. Storage replication (LRS in Budget, GRS in Enterprise) protects against hardware loss. Delta Lake's time-travel feature gives us the ability to query any table "as of" any point in the last 30 days, so accidental deletes or bad transformations can be rolled back. Beyond 30 days, point-in-time recovery is via the cold-storage archive (separate cost line, decide if needed).
