3. What went wrong

Root cause: Azure-side hostname-release lag. When the Classic frontend-endpoint www-hipyx-com was deleted at 22:14:52, Azure was expected to release the global hostname binding for www.hipyx.com within ~10-15 minutes (P50 historical) or up to ~20-25 minutes (P95). Tonight the lock was still active across all 8 retry attempts spanning 35+ minutes after deletion. Each az afd custom-domain create on the Standard profile returned the same error:

Code: Conflict
Message: We couldn't create your custom domain. Another custom domain
with the same host name already exists.

The "another custom domain with the same host name already exists" message comes from Azure detecting that the hostname is still bound at the AFD service level - even after the Classic frontend-endpoint resource is deleted, Azure's internal hostname-uniqueness lock persists for an asynchronous cleanup window that has no documented upper bound. Tonight's window exceeded all observed historical norms on this tenant.

This is not a defect in the cutover script, the network configuration, or the operator procedure. It is an Azure platform behavior that the script's retry loop is designed to absorb - and tonight it exceeded the conservative 45-minute budget the script provides.

6. Lessons learned

  1. Azure hostname-release lag has no upper bound. The 45-minute retry budget that we set tonight (significantly larger than historical P95 of 20-25 minutes) was still insufficient. For any future manual detach-then-attach cutover on AFD, the budget should be 90+ minutes, OR the manual approach should be replaced with the official atomic migration API where possible.
  2. The official atomic migration API (az afd profile migrate) does not have this failure mode. Azure handles the hostname transfer internally between the Classic and the new Standard profile. This was the path used successfully on 2026-04-24 for survey.farmboxrx.com. The constraint: az afd profile migrate creates a new Standard profile name and cannot merge into an existing one. So the constraint that drove tonight's manual approach (existing hipyx-std already had survey.farmboxrx.com) was the source of the failure mode.
  3. Auto-rollback design held up. The script's "fall through to rollback on exhausted retry budget" behavior worked exactly as specified. No manual intervention, no half-state, full restoration of Classic in 2 minutes. The rollback path is now battle-tested twice (today's earlier session + tonight's window).
  4. HTTPS managed cert on Classic survived the round-trip. Azure retained the cert binding through the deletion and re-creation, eliminating the cert-reissue worry that the script accounts for as a fallback. This is good news for any future rollback.
  5. Comms cadence worked. Approver was looped in early, DNS owner had a 1-2 hour check-in cadence (no need to glue her to a keyboard), incident channel was kept current. The opposite of how an unmonitored prod incident would feel.

Why the new approach has materially better odds:
  • Atomic API - Azure handles the hostname-uniqueness transfer internally as part of the migration; the manual delete-wait-recreate race we lost tonight does not exist in this path.

  • No 45-minute retry loop needed - Azure either accepts the migrate API call or returns an actionable error immediately.
