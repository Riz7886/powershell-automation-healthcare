Databricks Cold-Start & Serverless Audit

Workspace: pyx-warehouse-prod · Cluster: Processing Cluster · Audit prepared by Syed Rizvi

The Question

Why does the Processing Cluster in pyx-warehouse-prod take ~10 minutes to spin up? Can it be moved to serverless? What about the pyx-warehouse Pro SQL warehouse?

Bottom Line

Root Cause Identified

The cluster has no instance pool attached. Every restart provisions 3 fresh Azure VMs (1 driver + 2 workers minimum). VM provisioning + DBR image pull + Spark/Photon init + worker handshake from cold totals 7-10 minutes, exactly matching reported behavior.

Path Forward

Two fixes available, both low risk: (1) Promote the SQL warehouse to Serverless for ~10s start (one UI flip, no code change). (2) Move the all-purpose cluster to Serverless Compute for Notebooks for sub-30s start, OR attach an instance pool to drop cold start from ~10 min to ~1-2 min as a bridge.

SQL Warehouse — pyx-warehouse

Current state: Pro tier (visible in earlier screenshot). Pro warehouses cold-start in 3-5 minutes.

Recommended: Promote to Serverless. 5-10 second cold start, scales to zero when idle.

How: Workspace UI → SQL → SQL Warehouses → Edit pyx-warehouse → change type to Serverless → Save. No client change. No JDBC/ODBC string change. Same endpoint.

Concrete Recommendations

  1. Today (no risk, 30 seconds): Switch pyx-warehouse Pro → Serverless. Single UI flip. Drops cold start to ~10 sec for all SQL/BI users.
  2. This week (5 min verification + decision): In Processing Cluster → Advanced + Libraries tabs, confirm no init scripts, no custom Docker image, no legacy spark conf, no heavy cluster libraries. If clean: migrate the cluster to Serverless Compute for Notebooks (cold start < 30s).
  3. Bridge fix if Serverless migration is blocked: Create an Azure Databricks instance pool for Processing Cluster's worker type:
    • Pool type: Standard_D4ads_v6
    • min_idle_instances: 2 (matches the cluster’s minimum worker count)
    • preloaded_spark_versions: ["17.3.x-photon-scala2.13"]
    • idle_instance_autotermination_minutes: 60
    Attach Processing Cluster to the pool. Cold start drops from ~10 min to ~1-2 min.
  4. Going forward, scheduled jobs: Use Serverless Job Compute instead of all-purpose interactive clusters. Spins in seconds, billed per-second.
  5. Track cold-start as a SLI: The Databricks events API (/api/2.1/clusters/events) emits STARTING and RUNNING events with timestamps. Pull weekly, alert on any cluster averaging >= 5 min cold start in a 14-day window.

Cost Note

Serverless costs more per DBU than classic Pro / All-purpose. But:

  • Pro warehouse running idle inside its 30-min auto-stop window = paying for nothing useful
  • Serverless warehouse scales to zero in seconds = pay only for actual queries
  • For typical interactive / BI use, monthly all-in cost is usually similar or lower with Serverless because of utilization
  • For Processing Cluster on Serverless Compute: 3-VM cluster currently runs 6-22 DBU/h. Serverless equivalent is per-second billed only when notebook cells run, which for ad-hoc notebook work is dramatically less wall-clock

Recommended: 30-day side-by-side - leave one warehouse on Pro, promote one to Serverless, compare the DBU bills.
