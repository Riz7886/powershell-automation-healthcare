Azure Front Door Migration — Root Cause Report

Survey Portal (survey.farmboxrx.com) · Classic → Standard SKU
Prepared by Syed Rizvi · Production environment · 2026-04-23 22:00 to 2026-04-24 03:00

Executive Summary

The Classic Azure Front Door profile hipyx serving the survey portal was migrated to a new Standard SKU profile (hipyx-std) ahead of the Microsoft Classic AFD end-of-life. The new profile was provisioned with a dedicated endpoint, a managed TLS certificate for survey.farmboxrx.com, a Web Application Firewall policy in Detection mode, and a security policy binding the WAF to the custom domain.

Total elapsed time was approximately five hours, beginning at 22:00 and completing at 03:00 the following morning. Roughly two hours were the actual migration build. The remaining three hours were absorbed by issues external to the change itself: public Microsoft documentation referencing command syntax that is not implemented in the Azure CLI, PowerShell tooling behavior that silently corrupts Azure CLI invocations, cross-host text-encoding corruption during script transfer, and an undocumented Standard SKU WAF feature limitation that required a second architectural pass.

The Classic hipyx profile was not modified at any point during the work. The public survey portal served traffic normally throughout the migration and rollback is available at any time via a single DNS change.

Key Findings

Each item below represents time consumed by a gap that should not have existed. Times shown are approximate and reflect how the five-hour window actually broke down.

Finding Impact Time Lost
Microsoft's published migration guidance references the command az afd profile-migration, which does not exist in any version of the Azure CLI The documented command was rejected by the CLI. The real command surface is az afd profile migrate and az afd profile migration-commit — discovered by reverse-engineering the CLI help tree. ~45 min
PowerShell silently breaks Azure CLI output flag when az is wrapped in a function declaring [Parameter()] Declaring [Parameter()] elevates the wrapper to an advanced function, auto-injecting -OutVariable and -OutBuffer. Azure CLI then reports parameter name 'o' is ambiguous, pointing at the wrong code path. ~30 min
Cross-host clipboard transfer corrupted non-ASCII characters Em-dashes and smart quotes introduced by text editors survived visually but broke PowerShell parsing at the destination host. Required a full rewrite in pure ASCII, saved directly from Notepad. ~30 min
PowerShell here-strings require the closing marker at column zero Attempted a base64 here-string paste as a workaround. Parse error on line two with no guidance. The closing '@ must sit at column zero — undocumented edge case. Required abandoning the approach. ~20 min
Standard SKU silently rejects managed WAF rulesets Managed rule sets (OWASP Core, Bot Manager) are exclusive to Premium SKU. Standard accepts only custom rules. This is not prominent in the tier comparison documentation. Required a WAF re-design in place. ~30 min
Custom domain validation sequence is strict and non-parallelizable Managed certificate issuance is blocked until a TXT record at _dnsauth.<host> proves ownership. The validation token is only readable after the domain resource exists, so the sequence is fixed: create domain → read token → publish TXT → wait → publish CNAME. ~15 min
Custom domain names with dots are invalid as resource names The public hostname survey.farmboxrx.com must be stored as the resource name survey-farmboxrx-com. Invalid names return a generic validation failure with no indication that dots are the cause. ~10 min
Total time lost to undocumented or misdocumented issues ~3 h
Time spent on the actual migration build ~2 h

Engineering Decisions That Preserved Production

  • The Classic hipyx profile was not modified at any point. Every build step occurred on net-new resources so the live survey portal was never at risk.
  • Cutover was deliberately structured as a DNS change, not an Azure change, which places both the timing and the rollback under the network team's direct control.
  • The WAF was deployed in Detection mode first. This guarantees no false-positive user blocks during the initial observability window and gives the network team data before any move to Prevention mode.
  • An Azure-managed certificate was chosen over a customer-managed certificate to remove future renewal burden and avoid a Key Vault dependency.
  • The final script was made idempotent. Existing resources are detected and skipped, and 409 Conflict responses are treated as success. The script is safe to re-run at any time.

Information That Should Have Been Available Up Front

  • An authoritative, tested Microsoft runbook for the Classic → Standard migration that references the real command surface rather than the non-existent profile-migration syntax currently on the public docs.
  • An explicit pre-work decision on Standard versus Premium SKU. The WAF re-design phase was driven entirely by the absence of this decision going in.
  • A pre-arranged DNS change window aligned to the validation-token availability, so the managed certificate does not sit in Pending state while waiting for the DNS team.
  • A service principal with an appropriate token lifetime for long-running maintenance windows, in place of interactive user tokens that expire mid-session.

Recommendations

  1. Standardize on pure-ASCII scripts for any Azure CLI work on this tenant, saved directly from Notepad. Add to the team runbook as a coding standard.
  2. Prohibit wrapping az in a PowerShell advanced function. All Azure CLI calls must be invoked inline as & az ... 2>&1 to avoid the ambiguous-parameter class of failure.
  3. Revisit the Standard vs Premium WAF SKU decision in the next sprint. Standard gets us past the EOL deadline. Premium adds the managed OWASP ruleset.
  4. Retain the Classic hipyx profile in place for fourteen days after the DNS cutover to guarantee instant DNS-revert rollback. Deprovision only after the new profile has a clean two-week operating window.
  5. Save fx-complete.ps1 as the canonical reference implementation for any future AFD Standard build on this subscription. Every hard-won constraint from this session is baked into it.

Appendix A — DNS Cutover Package

Step 1. Publish TXT record. Required before the managed certificate can be issued.

Host  : _dnsauth.survey
Type  : TXT
Value : _rw7do2fh5k5xgevxze4vqn8gkh79v49
TTL   : 300

Step 2. Poll until the managed certificate moves from Pending to Approved.

az afd custom-domain show \
  --profile-name hipyx-std \
  --resource-group Production \
  --custom-domain-name survey-farmboxrx-com \
  --query domainValidationState -o tsv

Step 3. Once Approved, publish the CNAME. Traffic moves to the Standard profile after the TTL expires.

Host  : survey
Type  : CNAME
Value : pyx-fx-survey-ep-dthpdfh9ayb4htcm.zo2.azurefd.net
TTL   : 300

Appendix B — Rollback

Rollback is a DNS operation, not an Azure operation, and is available at any time with no downtime risk.

  1. In the farmboxrx.com DNS zone, delete or point the survey CNAME back to the Classic hipyx endpoint.
  2. Traffic returns to the Classic profile within the TTL window (300 seconds as configured).
  3. Leave the Standard resources in place for investigation. Idle AFD Standard billing is the only cost impact.
