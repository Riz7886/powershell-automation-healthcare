Datadog Health Audit - Runbook

One PowerShell script. Pulls all monitors, hosts, synthetic tests, and APM services from Datadog. Identifies anything in "No Data" state or otherwise not reporting. Generates an HTML report with full Databricks breakdown.

1. Prerequisites

  • Datadog API Key - Datadog UI » Personal Settings » API Keys » create or copy an existing one
  • Datadog Application Key - Datadog UI » Personal Settings » Application Keys » create one with at least monitors_read, hosts_read, synthetics_read, and apm_service_catalog_read scopes
  • Datadog Site - the URL fragment between app. and datadoghq.com in the address bar (e.g. us3, us5, eu). Default in script: us3.datadoghq.com
  • PowerShell 5.1 or 7+ on Windows. No additional modules needed - script uses built-in Invoke-RestMethod

2. Run the audit (interactive prompts for keys)

powershell.exe -ExecutionPolicy Bypass -File C:\Users\PC\Downloads\PYX-scripts\Datadog-Audit.ps1

Script will prompt for the API key + Application key (masked input). HTML report opens automatically when complete.

3. Run with keys passed in (no prompt)

powershell.exe -ExecutionPolicy Bypass -File C:\Users\PC\Downloads\PYX-scripts\Datadog-Audit.ps1 -ApiKey "<DD_API_KEY>" -AppKey "<DD_APP_KEY>"

4. Run with custom site / thresholds

powershell.exe -ExecutionPolicy Bypass -File C:\Users\PC\Downloads\PYX-scripts\Datadog-Audit.ps1 -Site "us3.datadoghq.com" -StaleMinutes 60 -NoTracesHours 8

6. Output

FileContents
datadog-audit-<timestamp>.htmlFull visual report - opens in browser, KPI cards + tables for No-Data monitors, stale hosts, paused synthetics, silent APM services, full Databricks breakdown
datadog-audit-<timestamp>.jsonSame data in machine-readable form for further analysis

7. Sections in the HTML report

  1. Monitors in No-Data state - the most common cause of UI tiles showing blank. Includes the metric query for each so root cause is visible at a glance.
  2. Hosts not reporting - Datadog Agent stopped checking in. Causes: agent service stopped, host shut down, network/auth break, decommissioned but not pruned.
  3. Paused synthetic tests - manually paused so producing no data while paused.
  4. APM services with no traces - registered in service catalog but no incoming traces in the window.
  5. Databricks monitors - full breakdown - every monitor tagged or named Databricks/DBX, with current state (Working / Alert / Warn / No Data) and the metric query so each one can be diagnosed individually.

8. Common root causes for "No Data" monitors

SymptomLikely causeFix path
Auto-created monitors all No DataCloud integration (AWS/Azure/GCP) lost API permission or was reconfiguredDatadog » Integrations » verify cloud account, re-authorize
Databricks monitors all No DataDatabricks integration broken or workspace metric forwarding disabledDatadog » Integrations » Databricks » check workspace tokens + system tables enabled
App Service / SQL monitors No DataAzure integration scope changed or sub removedDatadog » Integrations » Azure » verify subscription mapping
Custom-metric monitor No DataApplication stopped emitting the metric (deploy, refactor, env-var change)Read the Query column in the report - find that metric in Datadog Metrics Explorer to confirm it has stopped publishing
Single-host monitor No DataHost decommissioned but monitor still scoped to itUpdate monitor scope or delete
