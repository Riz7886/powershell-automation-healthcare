Datadog Audit + Fix - Runbook

Two-script workflow. Audit script identifies what's not reporting and why. Fix script auto-rotates Azure SP secret, restarts Datadog Agent on hosts via PSRemoting, schedules investigation downtimes on No-Data monitors. HTML reports for both.
Tip: Save this page as PDF via Ctrl+P -> "Save as PDF".

1. Files in this delivery

FilePurpose

Datadog-Fix.ps1Active fix. Auto-rotates Azure SP secret, PSRemoting Datadog Agent restart, downtimes No-Data monitors with investigating note.
Datadog-RUNBOOK.htmlThis document.

2. Prerequisites

  • Datadog API Key - Org Settings -> API Keys -> copy any existing key (suggest testing-auth for low blast radius)
  • Datadog Application Key - Org Settings -> Application Keys -> create or use one with broad read scope
  • Datadog Site - default us3.datadoghq.com
  • PowerShell 5.1+ or 7+ on Windows
  • Az PowerShell module + login - required only if using -RotateAzureSecrets on the fix script. Run Connect-AzAccount first; identity needs Application.ReadWrite.OwnedBy or equivalent on the App Registration.
  • PSRemoting from your workstation to the target hosts - required only if using -AgentHosts on the fix script

4. Fix script - usage

4a. Full-auto fix (recommended)

.\Datadog-Fix.ps1 -RotateAzureSecrets -MuteNoData -AgentHosts vm-moveit-auto,vm-moveit-xfr

Runs all auto-fixes: Azure SP secret rotation, agent restart on listed hosts, 7-day investigation downtime on every No-Data monitor.

4b. Preview without changing anything

.\Datadog-Fix.ps1 -RotateAzureSecrets -MuteNoData -AgentHosts vm-moveit-auto,vm-moveit-xfr -DryRun

Same script, but every action is reported as DRYRUN - no secret rotation, no agent restart, no downtime created.

4c. Diagnose only (no fixes applied)

.\Datadog-Fix.ps1

Pulls integration health + per-metric publish status. Reports only.

4d. Just restart the agents (no other actions)

.\Datadog-Fix.ps1 -AgentHosts vm-moveit-auto,vm-moveit-xfr

4e. Just clean the dashboard (downtime everything not reporting)

.\Datadog-Fix.ps1 -MuteNoData

4f. Just rotate Azure SP secret + push to Datadog

.\Datadog-Fix.ps1 -RotateAzureSecrets

5. Recommended workflow

  1. Run Datadog-Audit.ps1 first to confirm the picture (51 / 36 / 17 etc.)
  2. Send the audit HTML attachment + summary email to the requester
  3. Run Datadog-Fix.ps1 -RotateAzureSecrets -MuteNoData -AgentHosts <host1>,<host2> -DryRun first to preview
  4. If the dry-run report looks correct, drop -DryRun and run for real
  5. Wait 5-10 minutes for Azure metrics to start flowing again
  6. Re-run Datadog-Audit.ps1 - should show fewer No-Data monitors as data recovers
  7. For Databricks workspace + custom collector remediation, click into the integrations UI per links in the fix report
  8. Send the fix HTML to the requester as a follow-up

6. Common errors and fixes

SymptomCauseFix
403 on /api/v1/hosts or /api/v1/synthetics/testsApplication Key lacks scope or user role lacks infrastructure-read permissionScript wraps each section in try/catch - it'll skip and continue. Other sections still produce data.
"No Az PowerShell login"Az PowerShell not authenticatedRun Connect-AzAccount in the same PowerShell session before re-running the fix script
"App Registration not found in this tenant"Active Az tenant doesn't match the tenant configured in Datadog Azure integrationRun Connect-AzAccount -TenantId <the_tenant_from_dd_config> first
"Service 'datadogagent' not found" on a hostDatadog Agent not installed, or installed under a different service namePass -AgentServiceName "datadog-agent" for Linux-style installs, or install the agent first
WinRM "host unreachable" on PSRemotingWinRM disabled, or firewall blocks port 5985/5986Test-WSMan <host> to verify - if WinRM unavailable, restart agent via the host's own login (RDP/SSH) then unmute the monitor
Keys are sensitive. Do not paste API/App keys into chat, screenshots, commits, or logs. Use the interactive prompt (default) or set as environment variables and reference them in the script.
